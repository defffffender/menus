# План проекта: Платформа онлайн-меню для общепита (Узбекистан)

> Живой документ. Дополняем и меняем по ходу разработки.
> Последнее обновление: 2026-06-18

---

## 1. О проекте

SaaS-платформа для заведений общепита Узбекистана (рестораны, кафе, чайханы).
Заведения регистрируются, получают личный кабинет, создают онлайн-меню,
генерируют QR-коды для столов и принимают заказы прямо в кабинет.

**Ключевые принципы:**
- Гость **не регистрируется** — сканирует QR на столе, смотрит меню, делает заказ.
- Заказ автоматически привязан к **конкретному столу**.
- Заказы имеют **статусы** (Новый → Принят → Готовится → Подан → Закрыт).
- Сервис **не занимается оплатой счёта** — расчёт заведение делает само (наличные/терминал).
- Мультиязычность: **русский + узбекский + английский (ru/uz/en)** (отдельные поля в моделях).

## 2. Технологический стек

| Слой | Технология | Примечание |
|------|-----------|------------|
| Backend | Django 5.2.15, Python 3.12 | уже установлено в `venv` |
| Frontend | Django-шаблоны + **HTMX** + **Alpine.js** | server-side, без SPA |
| CSS / дизайн | **Tailwind CSS** | адаптив под телефон, темы под бренд |
| БД (dev) | SQLite | сейчас |
| БД (prod) | PostgreSQL | при деплое, переезд безболезненный |
| Картинки | Pillow | загрузка фото блюд |
| QR-коды | `qrcode` + Pillow | генерация в кабинете |
| SMS | **Eskiz.uz** | подтверждение +998; на старте — заглушка |
| Real-time | HTMX-опрос (polling) | без websockets на старте |
| Хостинг (prod) | VPS (Eskiz.uz / ahost.uz / Hetzner) | решим при деплое |

**Почему HTMX, а не React/SPA:** это полноценный продакшен-стек, не времянка.
Если в будущем кабинет упрётся в потолок — Django превращаем в API (DRF) и
добавляем React «островами», без переписывания всего сразу.

## 3. Архитектура: мультитенантность

Одна общая БД. У заведения (`Restaurant`) есть `owner`, а у всех остальных
моделей — `ForeignKey` на заведение. Кабинет всегда показывает только «свои»
строки. Отдельные БД/схемы на тенанта — не делаем (overkill).

## 4. Приложения Django

Все приложения живут в папке `apps/` (для визуального порядка) и
подключаются как `apps.<имя>`.

```
apps/core        — статические страницы (лендинг и т.п.)   [создано]
apps/accounts    — пользователи (вход по телефону +998)     [создано]
apps/restaurants — Заведение (тенант) + Сотрудники + Столы + QR
apps/menu        — Категории + Блюда + Варианты блюд
apps/orders      — Заказы + позиции заказа + статусы
```

> `restaurants`, `menu`, `orders` создаём на их этапах (вместе с моделями).

## 5. Модели (черновик)

### accounts
- **User** (кастомный) — вход по **телефону** `+998` вместо email.
  Завести **до первой миграции** (потом менять больно).

### restaurants
- **Restaurant** — `owner`, название, slug, лого, описание, адрес, телефон.
- **Membership** — связь (пользователь + заведение + роль). Гибкие роли.
- **Table** — `restaurant`, номер/название стола, `qr_token` (случайный,
  неугадываемый), активен ли.

### menu
- **Category** — `restaurant`, название (ru/uz/en), порядок сортировки.
- **Dish** — `category`, название (ru/uz/en), описание (ru/uz/en), **фото
  (обязательно)**, `is_available` (стоп-лист), доп. атрибуты: вес/объём,
  острота, время приготовления, метки.
- **DishVariant** — `dish`, название («обычный/большой»), **цена (целое,
  сумы)**. У блюда без вариантов — один вариант по умолчанию.

### orders
- **Order** — `restaurant`, `table`, **статус**, время создания, комментарий гостя.
- **OrderItem** — `order`, `dish_variant`, количество, **снимок цены и
  названия** на момент заказа (чтобы смена цены не «поехала» в старых заказах).

## 7. Пользовательские потоки

### Гость за столом (без регистрации)
1. Сканирует QR на столе → `/m/<qr_token>/`.
2. Видит меню заведения (фото, варианты, цены), собирает корзину.
3. Отправляет заказ → он привязан к этому столу, статус «Новый».
4. Видит экран «Заказ отправлен» + **статус внизу** (обновляется сам).
5. Может сделать **новый отдельный заказ** на тот же стол (дозаказ).

### Заведение (кабинет)
1. Регистрация → пользователь получает роль **Владелец** + пустой кабинет.
2. Владелец/управляющий создаёт: столы → категории → блюда (с фото/вариантами).
3. Генерирует QR на каждый стол → скачивает/печатает.
4. Принимает заказы, меняет их статусы. Новые заказы подсвечиваются (опрос).
5. Владелец приглашает сотрудников и назначает роли.

### Этап 7. Прод-архитектура и деплой (async, под масштаб 1000+ заведений)

**Целевая архитектура:**
```
nginx (TLS) → Uvicorn-воркеры (Django ASGI: HTTP-вьюхи + Channels-consumers)
PostgreSQL ← PgBouncer (пул)
Redis ── channel layer (WS-группы) + кэш + брокер Celery
Celery worker ── Eskiz SMS, превью картинок
Объектное хранилище (S3/R2) + CDN ── фото блюд
```

**Принципы:**
- **Мутации — по HTTP** (auth/CSRF/идемпотентность), **уведомления — по WebSocket** (push).
- HTTP-вьюхи остаются **синхронными** (ASGI гоняет в threadpool); async — только consumers. ORM на async НЕ переписываем.
- Каждый компонент конфигурится через env, без локального состояния → горизонтальное масштабирование без переписывания. Redis-channel-layer делает WS-группы общими между серверами.
- **Dev работает без новой инфраструктуры:** InMemory channel layer, Celery eager, SQLite, локальное хранилище. Прод переключается через env.

**Решения (зафиксированы):** realtime = HTML-фрагменты (HTMX over WS); фото = объектное хранилище + CDN; Celery + Redis — сразу.

**Подэтапы:**
- [x] **7.1 Конфиг/секреты:** `django-environ`, settings из env (`SECRET_KEY`, `DEBUG`,
      `ALLOWED_HOSTS`, `CSRF_TRUSTED_ORIGINS`, `REDIS_URL`). Dev-дефолты сохранены.
- [x] **7.2 Realtime (Channels):** ASGI-роутинг (`menus/asgi.py`), `CabinetOrdersConsumer`
      (auth+RBAC, группа `orders.<rid>`) и `GuestOrderConsumer` (`public_token`, `order.<id>`).
      Сервис-слой `apps/orders/services.py` (`place_order`/`apply_status_action` = запись +
      `transaction.on_commit` → `notify_order_change`). Лёгкий сигнал в группу → consumer
      рендерит фрагмент на своём языке и шлёт OOB-swap. Начальная отрисовка/ресинк при
      (ре)коннекте. HTTP-загрузка + медленный опрос (20с) — резерв на случай обрыва WS.
      Слой: InMemory в dev, Redis на проде (`REDIS_URL`). Сервер: daphne (ASGI).
      Проверено: ApplicationCommunicator (10/10) + сетевой смоук через Daphne.
- [ ] **7.3 Celery + Redis:** брокер, задачи (Eskiz SMS, ресайз фото). Dev — eager.
- [ ] **7.4 PostgreSQL:** `DATABASE_URL`, индексы (`Order(restaurant_id, status)`,
